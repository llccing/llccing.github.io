> 原文：[Can AI agents build real Stripe integrations?](https://stripe.com/blog/can-ai-agents-build-real-stripe-integrations)
> 作者：Michelle Bu
> 日期：2025 年 4 月 9 日

我们测试了 AI 智能体能否构建生产级别的 Stripe 集成——发现正确的工具和上下文至关重要。

在 Stripe，我们一直在思考 AI 智能体如何与 API 交互。随着智能体能力的不断增强，我们想要了解：它们真的能构建出真实、可用的 Stripe 集成吗？不是玩具示例，而是开发者会部署到生产环境的那种代码？

为了找到答案，我们进行了一系列实验。我们让 AI 智能体构建不同复杂度的 Stripe 集成——从简单的结账页面到多步骤的计费流程——并评估了结果。

## 我们学到了什么

简短的回答是：可以，但前提是有正确的配置。开箱即用的情况下，大多数 AI 智能体在 Stripe 集成方面表现不佳。它们会幻觉出 API 参数，使用已弃用的端点，生成看起来合理但实际上无法运行的代码。

但当我们为智能体提供了最新的 API 文档、可运行的代码示例以及 Stripe 自己的 SDK 时，结果有了显著改善。表现最好的智能体生成的代码与经验丰富的开发者编写的几乎无法区分。

## 实验

我们在一组集成任务中测试了多个领先的 AI 模型和智能体框架：

- 使用 Stripe Checkout 创建简单的支付页面
- 设置订阅计费流程
- 为市场平台构建 Connect 入驻体验
- 实现基于用量的计费与计量
- 处理支付生命周期事件的 Webhook

对于每个任务，我们从多个维度评估输出：代码能运行吗？错误处理正确吗？是否遵循了 Stripe 的最佳实践？代码审查者会批准吗？

## 什么让智能体成功

三个因素产生了最大的影响：

1. **获取最新文档。** Stripe 的 API 迭代很快。能够访问我们最新文档的智能体，比仅依赖训练数据的智能体生成了明显更好的代码。

2. **可运行的代码示例。** 能够参考真实、经过测试的集成示例的智能体犯的错误更少，遵循了更好的模式。

3. **SDK 感知。** 理解 Stripe 官方 SDK（而不是发起原始 HTTP 调用）的智能体生成了更简洁、更易维护的代码。

## 推出 Stripe Agent Toolkit

基于这些发现，我们构建了 [Stripe Agent Toolkit](https://github.com/stripe/agent-toolkit)——一套开源工具集，旨在帮助 AI 智能体构建更好的 Stripe 集成。

该工具包包括：

- **适用于主流智能体框架的工具**，如 LangChain、CrewAI、Vercel 的 AI SDK 和模型上下文协议（MCP）服务器
- **读写能力**，使智能体既能查询 Stripe 数据，也能创建资源
- **内置护栏**，限制智能体可以执行的操作，降低意外副作用的风险
- **计量计费支持**，使智能体能够代表你的客户报告用量

## 工作原理

该工具包为智能体提供了一种结构化的方式来与 Stripe API 交互。智能体使用类型化的工具来自动处理认证、验证和错误处理，而不是生成容易出错的原始 API 调用。

例如，使用该工具包的智能体可以：

- 创建客户并绑定支付方式
- 使用正确的价格和计费周期设置订阅
- 生成用于支付页面的 Checkout Session URL
- 记录用于计费目的的计量用量
- 监听和处理 Webhook 事件

每个工具都设计为自描述的，因此智能体能够理解需要哪些参数以及预期的输出是什么样的。

## 配置与安全

关键设计原则之一是安全性。默认情况下，工具包在受限模式下运行，智能体只能读取数据。写操作必须显式启用，你可以精确配置智能体被允许执行的操作。

这意味着你可以让智能体访问 Stripe，而不用担心它会意外创建收费或修改订阅。

## 下一步

我们将继续投入，使 Stripe 成为对智能体最友好的支付平台。这包括：

- 扩展工具包，增加更多工具和集成
- 改进面向 AI 消费的文档
- 构建评估框架来衡量智能体性能
- 与智能体框架开发者合作以确保兼容性

我们相信，AI 智能体将成为开发者与 API 交互的一种越来越重要的方式。通过提供正确的工具和上下文，我们可以帮助智能体构建不仅功能完备，而且可以投入生产的集成。
