# Project Guidelines

## Architecture

- This repo is a customized AstroPaper site built with Astro 4, Tailwind, and a small set of React islands. Prefer Astro components and pages by default; use React only for interactive islands such as [src/components/Search.tsx](src/components/Search.tsx), [src/components/Comments.tsx](src/components/Comments.tsx), and [src/components/RadioPlayer.tsx](src/components/RadioPlayer.tsx).
- Treat [src/config.ts](src/config.ts), [astro.config.ts](astro.config.ts), and [src/content/config.ts](src/content/config.ts) as the authoritative files for site metadata, locale, markdown behavior, and content schemas.
- TypeScript path aliases are defined in [tsconfig.json](tsconfig.json). Prefer `@components/*`, `@layouts/*`, `@utils/*`, and related aliases over long relative imports.

## Content Model

- Two content collections exist: `blog` and `short-stories`. Add or edit markdown in [src/content/blog](src/content/blog) and [src/content/short-stories](src/content/short-stories), following the schema in [src/content/config.ts](src/content/config.ts) and examples such as [src/content/blog/ai-changed-my-life.md](src/content/blog/ai-changed-my-life.md) and [src/content/short-stories/02-midnight-office.md](src/content/short-stories/02-midnight-office.md).
- `draft: true` hides content everywhere. In production, future posts are also hidden until `pubDatetime - SITE.scheduledPostMargin`; see [src/utils/postFilter.ts](src/utils/postFilter.ts). The site timezone is forced to Asia/Shanghai in [astro.config.ts](astro.config.ts), so date-related changes should be checked with that in mind.
- Blog posts and short stories share layouts and pagination utilities, but search and RSS are blog-only, while tag pages aggregate both collections. See [src/pages/search.astro](src/pages/search.astro), [src/pages/rss.xml.ts](src/pages/rss.xml.ts), [src/pages/tags/[tag]/index.astro](src/pages/tags/[tag]/index.astro), and [src/pages/tags/[tag]/[page].astro](src/pages/tags/[tag]/[page].astro).
- Post detail pages are generated by [src/pages/posts/[slug]/index.astro](src/pages/posts/[slug]/index.astro) and [src/pages/short-stories/[slug]/index.astro](src/pages/short-stories/[slug]/index.astro), both of which render [src/layouts/PostDetails.astro](src/layouts/PostDetails.astro).

## Build And Validation

- Prefer `pnpm` because the repo tracks [pnpm-lock.yaml](pnpm-lock.yaml). Core scripts are defined in [package.json](package.json): `pnpm dev`, `pnpm lint`, `pnpm format:check`, and `pnpm build`.
- `pnpm build` is the closest full validation because it runs `astro check`, builds the site, and then runs `jampack`. For a cheaper validation step, use `pnpm astro check` when you only need Astro and TypeScript verification.
- There is no dedicated unit or integration test suite in the repo today. Use the narrowest executable validation that matches the change.
- Docker-based local development exists in [docker-compose.yml](docker-compose.yml), but default to native package manager commands unless the task is specifically about container workflows.

## Conventions

- Treat [README.md](README.md) as upstream AstroPaper background, not the primary source of repo-specific behavior. Prefer the config, content, and route files above when guidance differs.
- Keep the current Chinese locale and publishing assumptions intact unless the task explicitly changes them. `LOCALE.lang` is `zh-CN`, many posts are Chinese-language, and timezone-sensitive logic assumes Asia/Shanghai.
- Preserve the existing route structure under [src/pages/posts](src/pages/posts), [src/pages/short-stories](src/pages/short-stories), [src/pages/radio](src/pages/radio), and [src/pages/shanghai](src/pages/shanghai). `radio` and `search` are hydrated pages; `shanghai` uses inline client-side DOM scripting.
- When adding interactivity, follow the current island pattern: render from Astro first, then hydrate only the interactive component instead of widening the React surface.
